## 测试总览

本项目包含两套单元测试：

| 项目 | 测试框架 | 测试运行命令 | 当前用例数 |
|------|----------|--------------|------------|

## 前端测试

### 测试框架

- **Vitest**：Vite 生态的测试运行器，速度快，兼容 Jest API。
- **jsdom**：模拟浏览器 DOM 环境，让 React 组件可以在 Node.js 中运行。
- **@testing-library/react**：以用户交互为中心的 React 组件测试工具。
- **@testing-library/jest-dom**：提供 `toBeInTheDocument()`、`toHaveStyle()` 等自定义断言。

### 测试配置

配置文件：`frontend/vitest.config.ts`

关键配置说明：

```typescript
export default defineConfig({
  plugins: [react()],                 // 使用 @vitejs/plugin-react 处理 JSX
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),  // 支持源码中的 @/services/api 别名
    },
  },
  test: {
    environment: 'jsdom',             // 模拟浏览器环境
    setupFiles: ['./src/test/setup.ts'],      // 全局测试前置文件
    globals: true,                    // 启用全局 describe/it/expect/vi
    testTimeout: 10000,               // 单条测试超时 10 秒
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html'],
    },
  },
});
```

### 测试文件

| 文件 | 用例数 | 说明 |
|------|--------|------|
| `frontend/src/services/api.test.ts` | 8 | 测试 `get` / `post` 的 URL 构造、headers、JSON body、错误处理 |
| `frontend/src/pages/index.test.tsx` | 9 | 测试首页标题、描述、链接、样式、列表项 |
| `frontend/src/pages/user.test.tsx` | 19 | 测试用户页加载状态、列表渲染、表单交互、提交、错误处理 |

### 运行方式

```bash
cd frontend

# 运行一次测试（推荐用于 CI/验证）
pnpm run test:run

# 监听模式，修改代码后自动重跑（推荐用于开发调试）
pnpm run test

# 生成覆盖率报告
pnpm run test:coverage
```

覆盖率报告会输出到 `frontend/coverage/` 目录。

### 测试覆盖范围

#### `api.test.ts`

- `get(path)`：
  - 正确拼接 `/api` 前缀并发起 GET 请求
  - 正确解析返回的 JSON
  - 非 2xx 状态码抛出包含状态码和状态文本的错误
- `post(path, body)`：
  - 正确设置 `Content-Type: application/json`
  - 正确序列化请求体
  - 非 2xx 状态码抛出错误

#### `index.test.tsx`

- 页面标题、项目介绍文本正确渲染
- 项目结构列表包含前端、后端、Nginx、Tomcat 四项
- “开始体验”区域和跳转链接存在且指向 `/user`
- 页面根容器使用了正确的内联样式
- 首页是纯展示组件，除链接外没有其他交互元素

#### `user.test.tsx`

- 组件挂载时自动调用 `GET /api/users`
- 加载状态显示“加载中...”
- 用户列表正确渲染，包含表头、表体、数据行
- 空列表时表格依然存在
- 表单允许输入姓名和年龄
- 提交时调用 `POST /api/users`
- 提交成功后清空表单并刷新列表
- 姓名或年龄为空时提示“请填写姓名和年龄”
- 获取用户失败或提交失败时弹出对应错误提示
- 包含返回首页的链接

## 后端测试

### 测试框架

- **JUnit 5**：Java 单元测试标准框架。
- **Spring Boot Test**：提供 `@SpringBootTest`、`@WebMvcTest` 等测试切片。
- **Mockito**：通过 `@MockBean` 模拟依赖。
- **MockMvc**：在不启动真实 Tomcat 的情况下测试 Spring MVC 控制器。

### 测试文件

| 文件 | 用例数 | 说明 |
|------|--------|------|
| `backend/src/test/java/com/example/demo/service/UserServiceTest.java` | 6 | 测试用户业务逻辑：初始化数据、查询、新增、ID 自增 |
| `backend/src/test/java/com/example/demo/controller/UserControllerTest.java` | 5 | 使用 `@WebMvcTest` 测试 REST API 接口 |
| `backend/src/test/java/com/example/demo/DemoApplicationTests.java` | 1 | Spring Boot 上下文加载冒烟测试 |

### 运行方式

```bash
cd backend

# 运行所有测试
mvn test

# 只运行某个测试类
mvn test -Dtest=UserServiceTest

# 只运行某个方法
mvn test -Dtest=UserControllerTest#getUserShouldReturnUser

# 跳过测试打包（不推荐，仅用于快速构建）
mvn clean package -DskipTests
```

### 测试覆盖范围

#### `UserServiceTest`

- 构造方法初始化 3 条默认用户数据
- `findAll()` 返回所有用户
- `findById(1L)` 返回对应用户
- `findById(999L)` 返回错误提示 `{error: "用户不存在"}`
- `create()` 分配自增 ID 并保存用户
- `create()` 不会修改调用方传入的原始 Map 对象

#### `UserControllerTest`

- `GET /api/users` 返回用户列表
- `GET /api/users/1` 返回指定用户
- `GET /api/users/999` 返回错误提示
- `POST /api/users` 创建用户并返回带 ID 的结果
- `GET /api/users/health` 返回健康检查信息

#### `DemoApplicationTests`

- 验证 Spring Boot 应用上下文可以正常加载
- 验证 `UserController`、`UserService`、`CorsConfig` 等 Bean 可被正确扫描

## 前后端同时运行（开发模式）

在开发新功能或调试时，通常需要同时启动前后端。本项目提供了 `scripts/run-dev.sh` 一键脚本：

```bash
cd /home/charles/code/nginxdemo
./scripts/run-dev.sh
```

### 脚本行为

1. 在后台启动后端：
   ```bash
   cd backend
   mvn spring-boot:run
   ```
   - 监听 `http://127.0.0.1:8081`
   - 上下文路径 `/backend`
   - 日志写入 `backend/backend-dev.log`

2. 通过 `GET http://127.0.0.1:8081/backend/api/users/health` 轮询，等待后端就绪。

3. 在前台启动前端 dev server：
   ```bash
   cd frontend
   pnpm run dev
   ```
   - 监听 `http://127.0.0.1:8000`
   - 通过 `frontend/.umirc.ts` 中的 proxy 把 `/api/*` 转发到后端

4. 按 `Ctrl+C` 退出前端 dev server 后，脚本会自动停止后台后端进程。

### 访问地址

| 地址 | 说明 |
|------|------|
| http://127.0.0.1:8000 | 前端页面 |
| http://127.0.0.1:8000/api/users | 经 Umi 代理的用户列表 API |
| http://127.0.0.1:8081/backend/api/users | 后端直连用户列表 API |
| http://127.0.0.1:8081/backend/api/users/health | 后端健康检查 |

### 手动分步运行

如果你不想使用脚本，也可以手动打开两个终端：

**终端 1：启动后端**

```bash
cd /home/charles/code/nginxdemo/backend
mvn spring-boot:run
```

等待看到类似日志：

```text
Started DemoApplication in x.xxx seconds
```

**终端 2：启动前端**

```bash
cd /home/charles/code/nginxdemo/frontend
pnpm install
pnpm run dev
```

打开浏览器访问 http://127.0.0.1:8000 即可。

### 与 Nginx + Tomcat 生产模式的区别

| 模式 | 前端 | 后端 | 代理 | 访问入口 |
|------|------|------|------|----------|
| 开发模式 | `pnpm run dev`（8000） | `mvn spring-boot:run`（8081） | Umi proxy | http://127.0.0.1:8000 |
| 生产/本地演示 | `frontend/dist` 静态文件 | 外置 Tomcat（8080） | Nginx | http://127.0.0.1:8088 |

开发模式下不需要启动系统 Nginx 和外置 Tomcat，调试更方便。

## 常见问题排查

### 前端测试报错 `Failed to resolve import "react/jsx-dev-runtime"`

**原因**：Vitest 需要直接依赖 `react` 和 `react-dom`，而 Umi 把它们放在自己的 node_modules 中，导致解析失败。

**解决**：已在 `frontend/package.json` 中显式声明 `react` 和 `react-dom` 为 dependencies，运行 `pnpm install` 即可。

### 前端测试报错 `Failed to resolve import "@/services/api"`

**原因**：Vitest 默认不识别 Umi 的 `@` 路径别名。

**解决**：已在 `frontend/vitest.config.ts` 中配置 `resolve.alias`，把 `@` 映射到 `frontend/src`。

### 前端测试报错 `Cannot access 'mockLink' before initialization`

**原因**：`vi.mock` 的工厂函数会被提升到文件顶部，不能引用外部变量。

**解决**：直接在 `vi.mock` 的工厂函数内部定义 mock 组件，而不是引用外部变量。示例：

```typescript
vi.mock('umi', () => ({
  Link: ({ to, children }: { to: string; children: React.ReactNode }) => (
    <a href={to}>{children}</a>
  ),
}));
```

### 后端 `DemoApplicationTests` 报端口冲突

**原因**：`@SpringBootTest` 会启动嵌入式 Tomcat，默认监听 `application.yml` 中的 8081 端口。

**解决**：

- 确保 8081 端口没有被其他进程占用。
- 或在 `backend/src/test/resources/application-test.yml` 中覆盖端口，并在测试类上加 `@ActiveProfiles("test")`。

### `scripts/test.sh` 报错 `pnpm: command not found`

**原因**：系统没有安装 pnpm。

**解决**：

```bash
corepack enable
corepack prepare pnpm@latest --activate
```

或在运行脚本前确保 `pnpm` 已在 PATH 中。

### 构建脚本想跳过测试怎么办

```bash
SKIP_TESTS=true ./scripts/build.sh
```

默认情况下 `build.sh` 会运行 Maven 测试，使用 `SKIP_TESTS=true` 可跳过以加速构建。
